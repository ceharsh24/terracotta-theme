Java-Specific Findings

Issue 1: Terracotta Keyword Overload

Java is keyword-heavy. On lines like public static final int, four consecutive tokens share the same terracotta color. Access modifiers (public/private), storage modifiers (static/final/volatile), primitive types (int/boolean/double), and control flow (if/return/try) ALL share #D06F50. This doesn't happen in Python or JS because those languages have fewer keywords per line.

Current: public static final int MAX = 1024;
Suggested: public static final int MAX = 1024;

Consider coloring primitive type keywords (int, double, boolean, void) as purple to match class type names and break up the orange runs.

Issue 2: Purple Collision — Annotations vs Types vs this

Three semantically distinct constructs share the exact same #AF85E6 italic:

Annotation: @Override
Interface: Transformer
this kw: this

In Java, annotations drive frameworks (Spring @Autowired, JUnit @Test). Consider: annotations → #B88EE4 bold italic, or this → terracotta italic (since it's a keyword).

Issue 3: Strings vs Numbers — Warm Gold Collision

String: "Terracotta" #DBAC3B
Number: 3.14159265 #C8A85C
Constant: MAX_CAPACITY #C8A85C

Both are warm gold. In Java, distinguishing string literals from numeric values and named constants is important for type safety. The difference is visible but can blur at a glance.

Missing: Java-Specific Token Rules

The theme has dedicated scopes for Python, Rust, Go, and Shell but nothing Java-specific. Recommended additions:

storage.type.primitive.java → #AF85E6 (purple)
storage.type.annotation.java → #B88EE4 italic bold
variable.language.this.java → #D06F50 italic

What Works Well

Functions (#3CB898 teal) pop beautifully against the warm background. Inherited types (#60A4E8 blue italic) are clearly distinct from declared types. Built-in constants (null/false pink italic) stand out — great for null-safety awareness. Operators (#6CA9D6 blue) provide clean visual separation. Enum members (LOW, CRITICAL) read as constants, which is semantically correct. The doc comment shade is subtly different from regular comments — a nice touch for Javadoc.
